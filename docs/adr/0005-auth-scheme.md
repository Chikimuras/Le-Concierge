# ADR 0005 — Auth scheme

- Status: Accepted
- Date: 2026-04-18
- Deciders: Alexandre Velia (solo)
- Tags: security, auth, rgpd

## Context and Problem Statement

Le Concierge is multi-tenant and handles PII + payment data. Authentication
is the gate between unauthenticated traffic and every downstream feature.
This ADR locks in the concrete scheme — algorithms, storage, and isolation
model — implemented by Phase 4a (foundations) and Phase 4b/4c (HTTP flows).

References: OWASP ASVS v4.0.3 §2 (authentication) and §3 (session
management); NIST SP 800-63B; CLAUDE.md §3.1; ADR 0002 (security baseline).

## Decision Drivers

- Solo developer: prefer well-trodden libraries, avoid custom crypto.
- Non-negotiable security posture (CLAUDE.md §3).
- Multi-tenant isolation must be enforced at multiple layers.
- Audit trail must survive attackers with DB write access, short of the
  signing key / hashing pepper.

## Considered Options

### Password hashing

1. **Argon2id (OWASP default)** — chosen.
2. bcrypt — older, slower to tune for modern GPUs, less memory-hard.
3. scrypt — memory-hard, but ecosystem less unified around parameters.

### Session storage

1. **Server-side sessions in Redis** — chosen.
2. Stateless JWT — rejected in CLAUDE.md §3.1 / §10 because revocation is
   not immediate.

### Pepper storage

1. **Env var / Docker secret, never DB nor TOML** — chosen.
2. DB column — defeats the purpose if the DB is compromised.
3. KMS / HSM — ideal long-term, overkill for Phase 4a.

### Multi-tenancy isolation

1. **Application layer (`org_id` in every query) + tests** — chosen for v1.
2. Postgres RLS policies — stronger defense-in-depth, adds schema complexity.
3. Per-tenant schema or database — hard to manage at solo-dev scale.

## Decision Outcome

### Password hashing: Argon2id

- Parameters: `m = 19 456 KiB (~19 MiB)`, `t = 2`, `p = 1`. Matches the
  OWASP 2024+ baseline; can be tightened later without a migration
  (Argon2 PHC strings embed the parameters, so rehashing happens lazily
  on next login).
- Salt: per-user, 16 random bytes, generated by `argon2::SaltString`.
- Pepper: application-wide secret loaded from `APP_AUTH__PEPPER`
  (env / Docker secret / SOPS). Minimum 32 random bytes
  (`openssl rand -hex 32`). Stored in Rust as
  `secrecy::SecretString` so it is redacted from `Debug` and zeroed on
  drop.
- Rejected: `argon2id_phc` without pepper (would miss the KDF-side
  secret and weaken the posture against DB-only leaks).

### Session storage: Redis

- Server-side sessions with opaque random IDs (32 bytes, CSPRNG).
  Stored under `session:<id>` in Redis with a TTL.
- Cookie: `HttpOnly; Secure; SameSite=Lax; Path=/` per CLAUDE.md §3.1.
  `SameSite=Strict` is used on the admin console (reserved for Phase
  4b+).
- Session ID rotation on every privilege elevation (e.g. after 2FA
  completes). OWASP ASVS 3.2.1.
- Rate limiting on `/auth/*`: 5 attempts / 15 min / IP, plus progressive
  account lockout (Phase 4b, via `tower-governor` + Redis counters).

### Pepper lifecycle

- Loaded at process startup; the API **fails closed** if
  `APP_AUTH__PEPPER` is unset.
- Rotating the pepper implies rehashing every password. Implementation
  plan: a future migration adds a `pepper_version` column; on login,
  if the stored password was hashed with an older pepper, we verify
  with that pepper, then rehash with the current one. Tracked as debt
  until the first rotation is needed.

### Multi-tenancy isolation: application layer for v1

- Every SQL query that touches tenant data accepts an `org_id` parameter
  and includes it in the `WHERE` clause. Type-level discipline via the
  `OrgId` newtype — mixing it with `UserId` is a compile error.
- Integration tests assert that data belonging to tenant A is not
  visible to a caller authenticated as tenant B (lands with the first
  tenant-scoped endpoint in Phase 4b+).
- Postgres RLS policies are **deferred**: they are the right
  defense-in-depth move but add operational complexity (per-request
  `SET LOCAL app.current_org`, policy bugs on every new table). We will
  revisit when the surface area justifies it — a later ADR will
  document the migration.

### Audit log: append-only, hash-chained

- Table `audit_events` (migration 0001):
  - `UPDATE`, `DELETE`, `TRUNCATE` blocked by triggers.
  - `hash = SHA-256(prev_hash || canonical_row_bytes)`.
  - Insertion happens under a Postgres advisory lock
    (`pg_advisory_xact_lock(…)`) so concurrent writers serialise and
    the chain stays contiguous.
- Payload is `jsonb` with a bounded `kind` column (≤ 64 chars).
- PII in payloads is minimised — emails are masked, no PAN/CVV/IBAN
  ever reach the log (ASVS 7.3).
- Hash chain verification is a CLI subcommand we plan to add in a
  later hardening phase; for now the primitive exists in the schema.

### Identifier scheme

- Both `UserId` and `OrgId` wrap a `uuid::Uuid` (v4, generated by
  `gen_random_uuid()` in Postgres).
- v7 (time-ordered) would give marginally better index locality but v4
  is sufficient at our expected volumes. Reconsider if hotspot writes
  on the primary key appear.

### Platform admin separation

- `platform_admins` is a dedicated table, not a `role` enum value. This
  avoids conflating "tenant role" (per-org) with "cross-tenant super
  user". Routes reserved for platform admins (`/admin`, CLAUDE.md §1)
  will check `platform_admins.user_id = session.user_id` rather than
  inspect a role enum.

## Validation

- Unit tests in `auth::hash` cover Argon2id round-trip, wrong-password
  rejection, and pepper sensitivity.
- Integration tests in `tests/auth.rs` exercise signup end-to-end
  against a real Postgres (testcontainers), including the
  `UPDATE`/`DELETE` blockers on `audit_events`.
- `sqlx::query!` macros catch schema/query mismatches at compile time.
